# Amazon DynamoDB

## Overview

Amazon DynamoDB is a fully managed, serverless NoSQL database service provided by AWS. It delivers high performance, low latency, automatic scaling, and built-in high availability, making it ideal for modern cloud-native and event-driven applications.

In this project, Amazon DynamoDB is used to store metadata of every file uploaded to Amazon S3 after it has been processed by the AWS Lambda function.

---

# Purpose in This Project

Amazon DynamoDB is responsible for:

- Storing uploaded file metadata.
- Maintaining a processing record for each uploaded file.
- Providing fast and scalable data retrieval.
- Acting as the metadata repository for the serverless pipeline.

# Why DynamoDB Was Used

Instead of storing file information inside the Lambda function or maintaining a traditional relational database, Amazon DynamoDB provides:

- Fully managed database service
- Automatic scaling
- Millisecond response time
- Serverless architecture
- High availability
- Zero infrastructure management

This makes DynamoDB an ideal choice for storing file metadata in an event-driven serverless application.

---

# Data Stored

Whenever a file is uploaded, Lambda stores the following information:

| Attribute | Description |
|------------|-------------|
| FileName | Name of the uploaded file |
| Bucket | S3 Bucket Name |
| FileSize | Size of the uploaded file |
| ContentType | MIME Type of the file |
| ETag | File checksum generated by S3 |
| UploadTime | Timestamp of processing |
| Status | Processing Status |

# How It Works

### Step 1

A user uploads a file into Amazon S3.

Example

```
employee-list.csv
```

---

### Step 2

Amazon S3 generates an ObjectCreated event.

---

### Step 3

AWS Lambda is triggered automatically.

---

### Step 4

Lambda retrieves metadata from the uploaded object including:

- File Name
- Bucket Name
- File Size
- Content Type
- ETag

---

### Step 5

Lambda inserts the metadata into the DynamoDB table.

Example

```python
table.put_item(
    Item=item
)
```

---

### Step 6

The metadata becomes immediately available for querying and future processing.

---

# Lambda Integration

AWS Lambda uses the Boto3 SDK to insert records into DynamoDB.

Example

```python
table.put_item(
    Item={
        "FileName": object_key,
        "Bucket": bucket,
        "FileSize": str(file_size),
        "ContentType": content_type,
        "ETag": etag,
        "UploadTime": upload_time,
        "Status": "Processed"
    }
)
```

---

# IAM Permissions Required

The Lambda execution role requires permissions to write data into DynamoDB.

Example IAM Policy

```json
{
  "Effect": "Allow",
  "Action": [
    "dynamodb:PutItem",
    "dynamodb:GetItem",
    "dynamodb:UpdateItem"
  ],
  "Resource": "arn:aws:dynamodb:ap-south-1:<AWS_ACCOUNT_ID>:table/FileMetadata"
}
```

# Interview Questions

### Why did you choose DynamoDB instead of Amazon RDS?

DynamoDB is a fully managed NoSQL database that offers automatic scaling, low latency, and seamless integration with AWS Lambda. It is ideal for serverless architectures where schema flexibility and high performance are required.

---

### Why was On-Demand Capacity Mode selected?

On-Demand mode automatically handles varying workloads without requiring capacity planning, making it cost-effective and suitable for development and serverless applications.

---

### Why is FileName used as the Partition Key?

Each uploaded file has a unique name, making it an efficient and simple primary key for identifying and retrieving metadata records.

Amazon DynamoDB serves as the metadata repository for this Serverless Event Processing Pipeline.
